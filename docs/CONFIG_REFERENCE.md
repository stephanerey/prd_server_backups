# Config Reference

Configuration files stay plain-text and manually editable.

## Main Files

- `/etc/server-backup/backup.conf`
  Main host-level configuration.
- `/etc/server-backup/targets.d/*.env`
  One target file per backup destination.
- `/etc/server-backup/profiles.d/*.conf`
  One profile file per backup scope.
- `/var/lib/server-backup/state/*.json`
  Last known operational state files.
- `/var/lib/server-backup/reports/*`
  Local text and JSON reports.

## backup.conf Fields

Validated required fields:

- `CONFIG_VERSION`
- `BACKUP_NAME`
- `RETENTION_DAILY`
- `RETENTION_WEEKLY`
- `RETENTION_MONTHLY`
- `LOCAL_DUMP_DIR`
- `LOG_FILE`
- `STATE_DIR`
- `REPORT_DIR`
- `RESTIC_CACHE_DIR`
- `RESTIC_PASSWORD_FILE`
- `RUN_RESTIC_CHECK`
- `RUN_PRUNE`

Retention policy fields:

- `RETENTION_DAILY`
- `RETENTION_WEEKLY`
- `RETENTION_MONTHLY`

For prune, each value may be `0` or a positive integer, but at least one of the three must be greater than `0`.

Validated when `EMAIL_REPORT_ENABLED="true"`:

- `EMAIL_REPORT_TO`
- `EMAIL_REPORT_FROM`
- `EMAIL_REPORT_COMMAND`

Supported `EMAIL_REPORT_COMMAND` values in this PR:

- `sendmail`
- `mail`

When `EMAIL_REPORT_COMMAND="sendmail"`, `server-backup` invokes:

```text
/usr/sbin/sendmail -t -f <EMAIL_REPORT_FROM>
```

The message still carries explicit `To:`, `From:` and `Subject:` headers.

Additional global fields generated by `server-backup setup`:

- `GENERATED_BY`
- `GENERATED_AT`
- `BACKUP_TAGS`
- `EMAIL_REPORT_SUBJECT_PREFIX`
- `EMAIL_REPORT_SEND_ON_SUCCESS`
- `EMAIL_REPORT_SEND_ON_FAILURE`
- `EMAIL_REPORT_ENABLED`
- `EMAIL_REPORT_TO`
- `EMAIL_REPORT_FROM`
- `EMAIL_REPORT_COMMAND`
- `RUN_COVERAGE_AUDIT`
- `COVERAGE_AUDIT_FAIL_ON_FAILURE`
- `COVERAGE_AUDIT_FAIL_ON_WARNING`

`BACKUP_TAGS` is split on whitespace at runtime. Example:

- `BACKUP_TAGS="pyparfums prod docker"`

becomes:

- `--tag pyparfums`
- `--tag prod`
- `--tag docker`

Default paths used by setup:

- `LOCAL_DUMP_DIR="/var/tmp/server-backup"`
- `LOG_FILE="/var/log/server-backup.log"`
- `STATE_DIR="/var/lib/server-backup/state"`
- `REPORT_DIR="/var/lib/server-backup/reports"`
- `RESTIC_CACHE_DIR="/var/cache/restic"`
- `RESTIC_PASSWORD_FILE="/etc/server-backup/secrets/restic-password"`

Health thresholds currently use built-in defaults:

- `BACKUP_MAX_AGE_HOURS=30`
- `RESTORE_TEST_MAX_AGE_DAYS=30`
- `COVERAGE_AUDIT_MAX_AGE_DAYS=7`
- `PRUNE_MAX_AGE_DAYS=14`

These thresholds are currently documented defaults, not mandatory
`backup.conf` keys.

## Target SFTP Fields

Required for every target:

- `CONFIG_VERSION`
- `TARGET_NAME`
- `TARGET_TYPE`
- `RESTIC_REPOSITORY`
- `RESTIC_PASSWORD_FILE`
- `RESTIC_CACHE_DIR`

Additional required fields for `TARGET_TYPE="sftp"`:

- `SSH_HOST_ALIAS`
- `SSH_HOSTNAME`
- `SSH_PORT`
- `SSH_USER`
- `SSH_IDENTITY_FILE`

Expected generated SFTP fields:

- `SSH_CONFIG_FILE`
- `SSH_KNOWN_HOSTS_FILE`

Generated SFTP repository format:

- `RESTIC_REPOSITORY="sftp:<alias>:<remote-path>"`

Restic commands also use an internal SFTP override derived from:

- `SSH_CONFIG_FILE`
- `SSH_HOST_ALIAS`

Equivalent runtime form:

```text
sftp.command="ssh -F /etc/server-backup/ssh/ssh_config <alias> -s sftp"
```

Local repository commands are serialized with a lock file:

- preferred: `/run/server-backup-repo.lock`
- fallback: `/tmp/server-backup-repo.lock`

Expected permissions:

- `/etc/server-backup/targets.d/<name>.env` : `0600 root:root`
- `/etc/server-backup/ssh/id_ed25519_<target>` : `0600 root:root`
- `/etc/server-backup/ssh/id_ed25519_<target>.pub` : `0644 root:root`
- `/etc/server-backup/ssh/ssh_config` : `0600 root:root`
- `/etc/server-backup/ssh/known_hosts` : `0600 root:root`

Validation now also checks:

- `SSH_PORT` numeric range
- existence of `RESTIC_PASSWORD_FILE`
- existence of `RESTIC_CACHE_DIR`
- existence of `SSH_IDENTITY_FILE`, `SSH_CONFIG_FILE`, `SSH_KNOWN_HOSTS_FILE` when accessible
- restrictive permissions on those files when accessible
- restrictive permissions on `RESTIC_PASSWORD_FILE` when accessible

Connectivity test command:

```bash
sudo server-backup target test <target>
```

This project now supports `repo init`, `repo check`, `repo snapshots`, `repo prune`, `backup run` and `restore test`.

## Profile Fields

Required for every profile:

- `CONFIG_VERSION`
- `PROFILE_NAME`
- `PROFILE_TYPE`
- `BACKUP_PATHS`

Optional:

- `EXCLUDES`
- `MISSING_PATH_BEHAVIOR`

Accepted `PROFILE_TYPE` values in this PR:

- `generic`
- `docker-host`
- `docker-app`
- `cis-site`
- `system-filesystem`

Recommended fields with warnings only:

- `DOCKER_INVENTORY` for `docker-host`, `docker-app`, `cis-site`
- `WEB_CONTENT_CRITICAL="true"` for CIS profiles
- `DATABASE_DUMPS` for CIS profiles
- `CONTENT_CLASSIFICATION` for CIS profiles

Additional profile fields used by the wizard:

- `DOCKER_INVENTORY`
- `WEB_CONTENT_CRITICAL`
- `APP_KIND`
- `CONTENT_CLASSIFICATION`
- `DATABASE_DUMPS`

`DATABASE_DUMPS` is a Bash-style array of semicolon-separated specs. Example:

```text
DATABASE_DUMPS=(
  "name=app-postgres;engine=postgresql;mode=docker;container=postgres;user=app;databases=appdb;globals=true;secret=/etc/server-backup/secrets/db/app/app-postgres.env"
)
```

Supported keys in each spec:

- `name`
- `engine`
- `mode`
- `container`
- `host`
- `port`
- `user`
- `databases`
- `all`
- `globals`
- `secret`

Supported engines in this PR:

- `postgresql`
- `mysql`
- `mariadb`

Supported modes in this PR:

- `docker`
- `local`
- `remote`

DB secrets are stored outside the profile in root-only files such as:

- `/etc/server-backup/secrets/db/<profile>/<name>.env`

Expected secret file content:

- PostgreSQL: `PGPASSWORD="..."`
- MariaDB/MySQL: `MYSQL_PWD="..."`

Expected permissions:

- secret directory: `0700 root:root`
- secret file: `0600 root:root`

Generated profile command:

```bash
sudo server-backup profile add
```

Profile file location:

- `/etc/server-backup/profiles.d/<name>.conf`

Runtime behavior:

- missing profile paths produce warnings
- a logical DB dump can be considered the primary coverage for a DB volume

## State Files

Common local state files:

- `/var/lib/server-backup/state/last-backup-run.json`
- `/var/lib/server-backup/state/last-prune-run.json`
- `/var/lib/server-backup/state/last-restore-test.json`
- `/var/lib/server-backup/state/last-coverage-audit.json`
- `/var/lib/server-backup/state/last-email-report.json`

These files are read by:

- `server-backup status`
- `server-backup operations status`
- `server-backup health`

## Reports

Common local reports:

- `backup-run-*.txt` / `backup-run-*.json`
- `prune-run-*.txt` / `prune-run-*.json`
- `restore-test-*.txt` / `restore-test-*.json`
- `coverage-audit-*.txt` / `coverage-audit-*.json`
- if all `BACKUP_PATHS` are missing, that profile fails and is skipped
- `EXCLUDES` entries become `restic backup --exclude <pattern>`

Current profile intent:

- `generic`: manually chosen application trees
- `system-filesystem`: broad system backup, not a logical DB dump replacement
- `docker-host`: host paths around Docker workloads
- `docker-app`: one Compose project and its mounts
- `cis-site`: CIS frontend/backend/content mapping with DB placeholder

## Sensitive Values

The CLI never displays values for keys containing:

- `PASSWORD`
- `SECRET`
- `TOKEN`
- `KEY`
- `PGPASSWORD`
- `MYSQL_PWD`
- `PASS`
- `PWD`

Sensitive values are shown as `<redacted>`.

## Sensitive Paths

- `/etc/server-backup/secrets`
- `/etc/server-backup/secrets/db`
- `/etc/server-backup/ssh`

## Runtime State

- `/var/lib/server-backup/state`
- `/var/lib/server-backup/reports`
- `/var/cache/restic`

Runtime files created by `backup run`:

- `/var/lib/server-backup/state/last-backup-run.json`
- `/var/lib/server-backup/reports/backup-run-YYYYMMDD-HHMMSS.txt`
- `/var/lib/server-backup/reports/backup-run-YYYYMMDD-HHMMSS.json`

Backup reports now also include logical DB dump results when `DATABASE_DUMPS` is configured on a profile.

Runtime files created by `repo prune`:

- `/var/lib/server-backup/state/last-prune-run.json`
- `/var/lib/server-backup/reports/prune-run-YYYYMMDD-HHMMSS.txt`
- `/var/lib/server-backup/reports/prune-run-YYYYMMDD-HHMMSS.json`

Runtime files created by `restore test`:

- `/var/lib/server-backup/state/last-restore-test.json`
- `/var/lib/server-backup/reports/restore-test-YYYYMMDD-HHMMSS.txt`
- `/var/lib/server-backup/reports/restore-test-YYYYMMDD-HHMMSS.json`

Runtime files created by `coverage audit`:

- `/var/lib/server-backup/state/last-coverage-audit.json`
- `/var/lib/server-backup/reports/coverage-audit-YYYYMMDD-HHMMSS.txt`
- `/var/lib/server-backup/reports/coverage-audit-YYYYMMDD-HHMMSS.json`

Runtime files created by DB dump tests:

- temporary directories under `/var/tmp/server-backup/db-dump-test-*`

Runtime files created by email reporting:

- `/var/lib/server-backup/state/last-email-report.json`

The state file records:

- report kind
- status
- recipient
- sender
- subject
- command used
- sent timestamp
- success or failure
- sanitized error message if delivery failed

Email bodies are sanitized before sending. Any line containing tokens such as
`PASSWORD`, `SECRET`, `TOKEN`, `KEY`, `PGPASSWORD`, `MYSQL_PWD`,
`RESTIC_PASSWORD`, `RESTIC_PASSWORD_FILE`, `SSH_IDENTITY_FILE`, `PRIVATE`,
or `PASSPHRASE` is replaced with `<redacted>`.

Coverage audit is local-only:

- no NAS access
- no restic command
- no Docker Compose execution
- no configuration rewrite

## Validation Commands

Validate installed configuration:

```bash
sudo server-backup config validate
```

Show a redacted view:

```bash
sudo server-backup config show
```

Run the global setup wizard:

```bash
sudo server-backup setup
```

Show the configured targets in status:

```bash
sudo server-backup status
```

Run a dry-run backup:

```bash
sudo server-backup backup run --dry-run
```

Run a prune dry-run:

```bash
sudo server-backup repo prune <target> --dry-run
```

Run a non-destructive restore test:

```bash
sudo server-backup restore test --target <target>
```
